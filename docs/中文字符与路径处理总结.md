# maya-pipeline-tools 中文字符与路径处理总结

## 项目背景

本项目是一套 Maya C++ 插件（`.mll`），运行在 Windows 中文环境下（系统 ACP 通常为 936/GBK）。所有源文件均为 C++，使用 Qt（QWidget）构建 UI，通过 Maya API（`MString`、`MGlobal`）与 Maya 交互。

核心挑战：**Windows 中文系统的默认代码页是 GBK (936)，而非 UTF-8 (65001)**。Maya 内部使用 UTF-16（`MString` 底层是 `wchar_t`），C++ 标准库的窄字符 API（`std::string`、`fopen`、`stat`）默认走系统 ACP。三者之间的编码不一致是所有问题的根源。

---

## 一、核心转换函数（在每个 .cpp 中重复定义）

项目在几乎每个源文件中都定义了以下 `static` 辅助函数，通过 `#ifdef _WIN32` 条件编译：

### 1. `toUtf8(const MString& ms) -> std::string`

将 Maya 的 `MString` 转为 UTF-8 编码的 `std::string`。

```cpp
// Windows 路径: MString -> wchar_t* -> WideCharToMultiByte(CP_UTF8) -> std::string
static std::string toUtf8(const MString& ms) {
    const wchar_t* wstr = ms.asWChar();
    if (!wstr || !*wstr) return std::string();
    int len = WideCharToMultiByte(CP_UTF8, 0, wstr, -1, nullptr, 0, nullptr, nullptr);
    if (len <= 0) return std::string(ms.asChar());  // fallback
    std::string result(len, '\0');
    WideCharToMultiByte(CP_UTF8, 0, wstr, -1, &result[0], len, nullptr, nullptr);
    if (!result.empty() && result.back() == '\0') result.pop_back();
    return result;
}
```

关键点：
- 使用 `ms.asWChar()` 而非 `ms.asChar()`。后者会按系统 ACP 转换，中文路径在非 GBK 环境下会乱码。
- fallback 到 `ms.asChar()` 仅作为最后手段。

涉及文件：`AnimExporter.cpp:71`、`BatchExporterUI.cpp:23`、`NamingUtils.cpp:19`、`PluginLog.cpp:40`、`RefCheckerUI.cpp:42`、`SafeLoaderUI.cpp:19`、`SceneScanner.cpp:32`

### 2. `utf8ToMString(const std::string& utf8) -> MString`

将 UTF-8 的 `std::string` 转为 `MString`，用于传递给 Maya API 执行命令。

```cpp
static MString utf8ToMString(const std::string& utf8) {
    if (utf8.empty()) return MString();
    int wlen = MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, nullptr, 0);
    if (wlen <= 0) return MString(utf8.c_str());
    std::wstring wstr(wlen, L'\0');
    MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, &wstr[0], wlen);
    return MString(wstr.c_str());
}
```

关键点：
- 走 `UTF-8 -> wchar_t -> MString(wchar_t*)` 路径，绕过 ACP。
- 如果直接 `MString(utf8.c_str())`，Maya 会按 ACP 解释字节，中文会乱码。

涉及文件：`AnimExporter.cpp:55`、`NamingUtils.cpp:36`、`PluginLog.cpp:31`、`pluginMain.cpp:18`、`RefCheckerUI.cpp:59`、`SafeLoaderUI.cpp:35`、`SceneScanner.cpp:49`

### 3. `utf8ToWide(const std::string& utf8) -> std::wstring`

将 UTF-8 字符串转为 `std::wstring`，用于调用 Windows 宽字符 API。

```cpp
static std::wstring utf8ToWide(const std::string& utf8) {
    if (utf8.empty()) return {};
    int wlen = MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, nullptr, 0);
    if (wlen <= 0) return {};
    std::wstring wstr(wlen, L'\0');
    MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, &wstr[0], wlen);
    if (!wstr.empty() && wstr.back() == L'\0') wstr.pop_back();
    return wstr;
}
```

涉及文件：`AnimExporter.cpp:42`、`ExportLogger.cpp:15`、`FileAnalyzer.cpp:17`、`PluginLog.cpp:19`、`SceneScanner.cpp:92`（内联）

### 4. Qt 桥接函数


## 二、文件 I/O：全部使用宽字符 API

项目中所有文件操作在 Windows 上都通过 `utf8ToWide()` 转换后使用宽字符版本的 API：

| 操作 | 窄字符（会失败） | 项目实际使用 |
|------|------------------|-------------|
| 打开文件 | `std::ifstream ifs(path)` | `std::ifstream ifs(utf8ToWide(path))` |
| 写入文件 | `std::ofstream ofs(path)` | `std::ofstream ofs(utf8ToWide(path))` |
| 文件是否存在 | `stat(path, &st)` | `_wstat(utf8ToWide(path).c_str(), &st)` |
| 获取文件大小 | `stat(path, &st)` | `_wstat(utf8ToWide(path).c_str(), &st)` |
| 创建目录 | `mkdir(path)` | `_wmkdir(utf8ToWide(path).c_str())` |
| 重命名文件 | `rename(a, b)` | `_wrename(wA.c_str(), wB.c_str())` |
| 删除文件 | `remove(path)` | `_wremove(wpath.c_str())` |
| 目录遍历 | `FindFirstFileA` | `FindFirstFileW` + `wstring` |

典型示例（`AnimExporter.cpp:596-604`）：

```cpp
static bool fileExistsOnDisk(const std::string& path) {
#ifdef _WIN32
    struct _stat st;
    return (_wstat(utf8ToWide(path).c_str(), &st) == 0 && (st.st_mode & S_IFREG));
#else
    struct stat st;
    return (stat(path.c_str(), &st) == 0 && (st.st_mode & S_IFREG));
#endif
}
```

MSVC 的 `std::ofstream` 支持 `std::wstring` 构造函数，这是项目能直接传 `utf8ToWide(path)` 给 `ofstream` 的前提（`PluginLog.cpp:131-137`）。

## 三、MEL/Python 命令执行中的编码处理

### MEL 命令

所有 MEL 命令字符串都通过 `utf8ToMString()` 转换后传给 `MGlobal::executeCommand()`：

```cpp
// AnimExporter.cpp:194
MStatus status = MGlobal::executeCommand(utf8ToMString(cmd));
```

这确保了命令中包含的中文路径（如节点名、文件路径）不会被 ACP 破坏。

### Python 命令（SafeOpenCmd 的教训）

`SafeOpenCmd.cpp:20-27` 记录了一个重要的经验教训：

```cpp
// We use Python cmds.file() instead of MEL eval+string concatenation
// because MEL string interpolation corrupts Chinese/Unicode paths on
// non-UTF-8 Windows systems (ACP=936), causing Maya to crash when
// opening .mb files.  Python passes the path as a proper argument
// without re-encoding.
```

MEL 的字符串拼接在处理中文路径时会导致编码损坏，甚至导致 Maya 崩溃。解决方案是改用 Python `cmds.file(path, open=True, ...)`，让 Python 直接传递路径对象，避免 MEL 层的字符串重编码。

### Python 代码中的编码

`RefCheckerUI.cpp:76` 中执行 Python 代码时也使用 `utf8ToMString`：

```cpp
// Uses utf8ToMString to correctly handle Chinese/Unicode in the Python code.
MString mpy = utf8ToMString(pyCode);
MGlobal::executePythonCommand(mpy);
```

## 经验教训总结

### 必须做的

1. **内部统一使用 UTF-8**：所有 `std::string` 约定为 UTF-8 编码，这是整个项目的编码契约。
2. **MString 转换必须走 wchar_t**：`MString::asWChar()` + `WideCharToMultiByte(CP_UTF8)` 是唯一可靠路径。绝不能用 `MString::asChar()`（它走 ACP）。
3. **文件 I/O 必须用宽字符 API**：`_wstat`、`_wmkdir`、`_wfopen`、`std::ofstream(wstring)` 等。窄字符版本在中文路径下会失败。
4. **日志文件写 UTF-8 BOM**：Windows 编辑器（记事本、VS Code 等）依赖 BOM 自动检测 UTF-8。
5. **源文件必须保存为 UTF-8**：配合 `u8""` 字面量使用，否则中文字符串在编译时就已损坏。

### 绝不能做的

1. **不要用 MEL 拼接中文路径**：MEL 字符串插值会破坏 Unicode 编码，改用 Python `cmds.file()` 直接传参。
2. **不要在工作线程中使用 QDir**：Maya 环境下 Qt 的某些类在非主线程中会崩溃，改用 Win32 原生 API。
3. **不要假设系统是 UTF-8**：即使 Windows 10+ 有 UTF-8 ACP 选项，绝大多数中文 Windows 系统仍然是 GBK (936)。
4. **不要用 `MString(const char*)` 传递 UTF-8 字符串**：这个构造函数按 ACP 解释输入，必须先转 `wchar_t*` 再构造。

### 可以改进的

1. **消除重复代码**：`toUtf8`、`utf8ToMString`、`utf8ToWide` 在 7+ 个文件中重复定义，应提取到公共头文件。
2. **考虑 C++17 `std::filesystem`**：如果编译器支持，`std::filesystem::path` 在 MSVC 上原生支持 UTF-8（需配合 `std::filesystem::u8path` 或 C++20 的 `char8_t`），可以简化部分路径操作。
3. **统一错误处理**：Win32 API 调用（`MultiByteToWideChar` 等）的返回值检查模式可以封装为 RAII 或异常。
